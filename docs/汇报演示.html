01 · Why

居民很少说「我要路灯」，但会说「晚上太黑」

政务平台上的诉求是有意识投诉； 社交媒体上的抱怨往往是无意识表达。 中间存在断裂：识别到抱怨 ≠ 转化为可行动的公共服务需求。

本项目切口：把「沉默抱怨」翻译成「沉默需求」。

对比一眼看懂

  • 12345 / 留言板：主动、结构化、已是诉求
  • 微博日常帖：碎片、情绪化、未必自觉在「提需求」
  • 我们的输出：缺口清单 + 优先级，辅助决策而非替代决策

怎么采、怎么洗

  • 关键词 taxonomy（设施 × 症状 × 地点）
  • crawl4weibo 真实抓取（非样例）
  • 去重 / 过短过滤；区县与地标推断

汇报时要主动说的局限

  • 公交站/天桥保留极少（仅 7 条）→ 结论勿外推
  • 有区县定位约 15% → 空间分析需「未定位/全市」桶
  • 仍有残余噪声，由 Lab2 证据门控消化
05 · Lab2 核心贡献

不是正负情感，而是「抱怨→需求」词典

「广顺南大街路灯四天不亮，下雪摸着黑，危险！」
→
lighting_repair
道路照明维护与夜间步行安全
意图=建议/抱怨 · 情绪=焦虑 · 高紧迫度
「燃油车赖在充电桩车位不走，害得排队充不了」
→
charging_occupancy_governance
充电车位占位治理

这就是 domain adaptation：商品评论的正负分类回答不了「该修什么、优先修哪里」。

08 · 到决策的最后一公里

Lab3：把映射结果变成行动清单

给决策者的不是情绪仪表盘，而是缺什么、在哪、先修谁。

输入 Lab2 的 analyzed JSONL → 按区县×需求聚合 → 紧迫度排序 → 情况简报 / 缺口清单。

当前可汇报的决策信号（基于 Lab2）

  • 最高频：道路照明维修（508）
  • 其次：公共充电供给（276）与照明补点（177）
  • 占位治理：样本量小但问题尖锐（22）
  • 公交/天桥：数据不足，标为待补采
09 · Takeaways

三句话收束

1. 问题定义对了

沉默需求 = 社媒无意识抱怨 → 可执行公共服务需求，区别于政务留言与商品评论。

2. 真数据跑通了

1500 微博 → 1473 清洗 → 66.9% 需求映射；禁止假数据作为底线。

3. 适配在标签体系

贡献不在「更炫的模型」，而在意图+情绪+需求词典的领域翻译层。

诚实局限（老师爱听）

公交样本过少；地理定位覆盖有限；Lab3 生成式报告仍可加强权威知识 grounding。下一步：补采 transit、提高 POI 覆盖、用 LLM 生成带证据的简报。
