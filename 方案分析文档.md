# 方案分析文档：公共资源「沉默需求」发现

> 分析对象：`公共资源的“沉默需求”发现：为什么重要、具体做什么.md`  
> 对照依据：《课程实践任务书》及课程对领域适配、流水线完整性的要求

---

## 1. 方案一句话概括

居民很少会发帖说「我要某个公共设施」，但会抱怨「楼下太吵」「孩子没处玩」「晚上路太黑」。本方案的核心是：**从社交媒体上的无意识抱怨中，识别并翻译出未满足的公共服务需求**，最终输出可供城市管理者使用的决策支持产品（热力图、缺口清单、优先级建议）。

整体技术链路与课程三模块一一对应：

```text
原始社媒数据 → 预处理与空间/实体标注
            → 意图分类 + 观点/情感分析 + 需求映射
            → 生成式决策支持（报告 / 清单 / 问答）
```

## 2. 与课程任务书的对齐度

| 任务书要求 | 本方案对应设计 | 对齐评价 |
| --- | --- | --- |
| 选定具体应用领域 | 城市公共资源 / 公共服务设施治理 | 明确，非泛舆情 |
| Lab 1 数据采集与清洗 | 设施词 + 动作词爬取；清洗与 POI/地点处理 | 对齐良好，需补 taxonomy 叙事 |
| Lab 2 情感/观点分析 | 抱怨/建议/询问三分类 + NER + 需求映射 | 强领域适配，优于通用正负情感 |
| Lab 3 生成式决策支持 | 热力图、缺口清单、优先级排序与措施建议 | 决策产品形态清楚 |
| 统一数据格式衔接 | 三阶段天然串成流水线 | 需明确各模块中间 JSON/表结构 |
| 强调领域特征与通用场景差异 | 政务平台显性诉求 vs 社媒无意识抱怨 | 研究 gap 讲得清楚 |
| 不强制复杂算法 | 爬虫/BERT/NER/LLM，成熟技术可复用 | 符合课程实践目标 |

**总评：高度契合课程项目目标。** 选题不是「做一个情感分类 Demo」，而是「做一条从社会感知到决策支持的完整链路」，汇报时故事完整、可演示空间大。

## 3. 方案优势分析

### 3.1 研究切口有辨识度

「沉默需求」比「舆情监测」更精准：

- 传统工作多停在识别抱怨或情绪；
- 政务平台（留言板、12345）上的诉求本身已是有意识表达；
- 本方案关注的是社媒中**尚未被正式提出、但真实存在**的需求。

这直接回应了老师强调的问题：**你们的领域和通用场景差在哪里？**

### 3.2 Lab 2 的领域特化设计是亮点

若只做正/负/中，会像商品评论情感分析。本方案改为：

1. **意图识别**：抱怨 / 建议 / 询问  
2. **实体抽取**：设施类型、地点  
3. **需求映射**：抱怨 → 可执行的公共服务需求  

其中「需求映射词典」是方案的核心创新点，也是汇报时最容易讲清楚的技术贡献。

### 3.3 Lab 3 成果可感知、可演示

决策支持输出不是空泛对话，而是：

- 需求热力图  
- 设施缺口清单  
- 建设/治理优先级  
-（可选）智能问答或情况简报  

符合任务书对「为决策者提供辅助分析，而非替代人工决策」的定位。

### 3.4 四天时间窗内可实施

技术路线成熟，不依赖自研复杂模型；优秀成果还可继续打磨为论文题材（城市治理 + 社会计算 + 生成式决策）。

## 4. 主要风险与不足

### 4.1 范围过大（最高优先级风险）

当前关键词覆盖公园、充电桩、人行道、天桥、路灯、垃圾桶、公交站、厕所、健身器材、图书馆、菜市场等。四天内若全面铺开，容易变成浅而碎的系统。

**建议收敛：**

- 选定 **1 个城市**（如北京）；  
- 选定 **2～3 类设施/问题主题**，例如：  
  - 道路照明与夜间步行安全  
  - 儿童活动 / 公共活动空间噪声冲突  
  - 公共充电设施可得性  

先做深一条可演示闭环，再扩展类型。

### 4.2 关键词设计缺少「taxonomy」闭环说明

任务书明确要求思考：

- 关键词为何这样选；  
- taxonomy 是什么；  
- 领域事件特征如何影响采集策略。

当前方案有词表，但论证还不够结构化。建议补成四层 taxonomy：

```text
设施类型（What）
  └─ 问题表现（How wrong）
       └─ 地点/POI（Where）
            └─ 需求暗示句式（Implicit need）
```

并补充领域特征叙事：公共设施抱怨多为**慢性、地域性强、非突发、持续积累**——与自然灾害的突发性、阶段性不同，因此更适合「连续采样 + 空间聚合」，而非「事件触发式抓取」。

### 4.3 数据采集不稳定

微博 / 小红书反爬与登录门槛高，真实风险大。老师也允许：采不到仍可用公开数据，重点看 pipeline 思想。

**预案：**

1. 主路径：限定平台 + 限定主题关键词采集；  
2. 备用路径：公开投诉/留言开放数据 + 少量社媒样例；  
3. 最低可行：人工精标样例数据集，保证 Lab 2 / Lab 3 可跑通并演示。

### 4.4 地理信息依赖过强

方案依赖 POI/地理标签做热力图，但大量帖子没有可靠 geo。

**需要 fallback：**

- 文本地点 NER（商圈、小区、道路名）；  
- 用户简介/发帖上下文中的地点线索；  
- 无法定位时进入「全市按设施类型聚合」桶，不强行画假点。

### 4.5 Lab 2 若完全放弃「情感」会偏题

课程 Lab 2 标题是「情感 / 观点分析」。意图分类很好，但汇报时最好叠加一层情绪信号（不满、焦虑、期待、中性），用于 Lab 3 的紧迫度加权。

推荐标签体系（示例）：

| 维度 | 标签 |
| --- | --- |
| 意图 | 抱怨 / 建议 / 询问 / 其他 |
| 情感 | 不满 / 焦虑 / 期待 / 中性 |
| 实体 | 设施类型、地点 |
| 需求 | 映射后的标准需求类目 |

### 4.6 权威知识融合偏弱

第二天课程强调：不能只依赖社交媒体，需要权威知识融合。当前方案有映射规则，但尚未引入外部参照（如设施配置标准、服务半径、规划规范、公开统计）。

建议在 Lab 3 增加：

- 社媒需求信号；  
- 权威标准 / 指南；  
- 生成「证据—缺口—建议」结构化简报。  

这能显著提高可信度，也更贴「知识融合与决策依据构建」。

## 5. 与通用方案对比：真正的 domain adaptation 在哪里

| 对比项 | 通用情感分析 / 商品评论 | 本方案（沉默需求） |
| --- | --- | --- |
| 文本性质 | 明确评价对象与态度 | 日常抱怨，未必自觉是「诉求」 |
| 分类目标 | 正/负/中 | 意图 + 情绪 + 需求类目 |
| 输出用途 | 品牌声量、满意度 | 公共服务缺口与治理优先级 |
| 空间维度 | 通常不需要 | POI / 区域聚合几乎必需 |
| 决策形态 | 仪表盘指标 | 可执行设施清单与行动建议 |
| 知识融合 | 可选 | 应引入规划/设施配置标准 |

**结论：方案的差异化不在于算法更新，而在于问题定义与标签体系。** 这正是课程希望看到的「领域适配」。

## 6. 建议的中间数据契约（保证三天可衔接）

为实现「统一数据格式」，建议约定一条主记录结构（示意）：

```json
{
  "id": "post_001",
  "platform": "weibo",
  "text": "望京这边晚上路灯坏了好多，走路好害怕",
  "time": "2026-07-10T21:13:00+08:00",
  "geo": {
    "raw": "望京",
    "poi": null,
    "district": "朝阳区",
    "confidence": 0.7
  },
  "clean_text": "...",
  "intent": "抱怨",
  "emotion": "焦虑",
  "entities": {
    "facility": ["路灯"],
    "location": ["望京"]
  },
  "mapped_need": "道路照明维护/夜间步行安全保障",
  "urgency_score": 0.82,
  "evidence_count_group": 17
}
```

- Lab 1 产出：清洗后的 `id/text/time/geo/...`  
- Lab 2 补齐：`intent/emotion/entities/mapped_need`  
- Lab 3 消费：按区域与需求类目聚合，再生成报告 / 看板 / 问答

## 7. 四天落地建议（可执行拆分）

### Day 1（Lab 1）

- 确定城市 + 2～3 类设施主题  
- 完成关键词 taxonomy（设施 / 表现 / 地点 / 暗示句式）  
- 跑通采集或导入备用数据集  
- 完成基础清洗：去重、去噪、短文本过滤、基础地点抽取  

### Day 2（Lab 2）

- 标注/构建小样本训练集（可弱监督 + 人工校正）  
- 实现意图分类 +（可选）情感分类  
- 实现设施/地点 NER  
- 固化「抱怨 → 需求」映射词典（v1）  

### Day 3（Lab 3）

- 区域与需求类目聚合  
- 紧迫度打分（频次 × 情绪强度 × 影响面代理指标）  
- LLM 生成：情况简报 / 缺口清单 / 优先级建议  
- 引入少量权威知识片段做 grounding  

### Day 4（整合与汇报）

- 三模块联调  
- 准备 demo：样例帖 → 映射结果 → 报告页  
- Slides：背景 → 架构 → Lab1/2/3 → 演示 → 局限与下一步  

---

## 8. 汇报时建议突出的四句话

1. **问题**：城市治理缺少对「未正式投诉、但真实存在」的公共服务需求的主动感知能力。  
2. **方法**：用社会计算从社媒抱怨中识别意图、实体与情绪，再映射为标准需求。  
3. **差异**：不同于政务平台显性诉求分析，也不同于商品评论情感分类。  
4. **产出**：面向决策者的缺口清单与优先级建议，人机协同、不替代决策。
